> 插入目的：承接 2.1 内存四区中"栈帧"的概念。递归 = 函数调用自身 = 栈帧不断压入，直到终止条件触发逐一弹出。学生已经知道栈帧是什么，递归是"亲眼看到栈工作"的最佳案例。

| #   | 卡类型   | 标题                         | 内容要点                                                                                                                                                                                            |
| --- | -------- | ---------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| c1  | concept  | 递归是什么                   | 函数内部调用自己。两个要素：①终止条件（递归的"出口"，没有它就会无限递归→栈溢出）；②递推关系（把大问题拆成规模更小的同类问题）。经典比喻：俄罗斯套娃——打开一个里面还有一个，直到最小的那个           |
| c2  | concept  | 递归与栈帧——回到 2.1         | 每次函数调用都会在栈上压入一个新的栈帧。递归 5 层 = 5 个栈帧压在栈上。终止条件触发后逐一弹出返回。结合 2.1 的栈帧知识：递归不是魔法，就是普通的函数调用——只是调的是自己                             |
| c3  | code     | 阶乘 — 递归入门              | `int factorial(int n) { if (n <= 1) return 1; return n * factorial(n-1); }` 拆解 factorial(4) 的完整调用过程：4→3→2→1(终止)→逐层返回。程序 + 调用链示意图                                           |
| c4  | code     | 斐波那契 — 双递归            | `int fib(int n) { if (n <= 1) return n; return fib(n-1) + fib(n-2); }` 展示 fib(5) 的递归树。指出重复计算问题（fib(3) 被算了两次），引出"递归优雅但可能低效"                                        |
| c5  | concept  | 递归 vs 循环                 | 递归优势：代码简洁、天然适合树/图的遍历。循环优势：不额外占用栈空间、通常更快。选择标准：问题本身是递归定义的就用递归（如树的遍历），简单重复操作用循环。尾递归优化简介——编译器能把尾递归优化成循环 |
| c6  | practice | 递归没有终止条件会发生什么？ | choice: 编译错误 / 无限递归直到栈溢出 / 自动返回0 / 只执行一次 → 无限递归直到栈溢出（每次调用都压栈，栈空间耗尽时程序崩溃）                                                                         |

### 4.4 拷贝控制 (7 卡)

> 从构造析构拆分出来独立成节点。Rule of Three 是 C++ 最独特也最关键的概念——Java/Python 没有深拷贝问题，因为对象默认就是引用语义。C++ 的值语义 + 默认浅拷贝是所有指针 bug 的根源。加入动画。

| #   | 卡类型        | 标题                                     | 内容要点                                                                                                                                                                                      |
| --- | ------------- | ---------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| c1  | concept       | 编译器偷偷帮你写了什么                   | 默认构造、拷贝构造、拷贝赋值、析构——不写时编译器自动生成。**自动生成的拷贝是浅拷贝**（逐成员复制），指针成员复制的是地址值，两个对象指向同一块堆内存                                          |
| c2  | concept       | 浅拷贝的灾难                             | `Buffer a(10); Buffer b = a;` — 两个 data 指针指向同一块堆内存。一个析构释放 → 另一个的 data 变成悬空指针 → 访问崩溃 / 第二个析构时 double free                                               |
| c3  | **animation** | **浅拷贝 vs 深拷贝**                     | MemoryBox 三步：(1) 对象 a 的 data 指向堆区 (2) 浅拷贝后两个 data 指向同一内存 (3) a 析构释放 → b 的 data 悬空。深拷贝：各自拥有独立的堆区副本                                                |
| c4  | concept       | Rule of Three                            | 如果自定义了析构函数、拷贝构造或拷贝赋值中的**任意一个**，三个都需要自定义。原因：自定义析构 ≈ 在管理资源 → 资源需要正确的拷贝语义。编译器生成的默认版本只会做浅拷贝                          |
| c5  | code          | 深拷贝实现                               | `Buffer(const Buffer& b) : len(b.len) { data = new char[len]; memcpy(data, b.data, len); }` + 拷贝赋值运算符（**必须检查自赋值** `if(this != &other)`，然后释放旧资源、分配新资源、拷贝数据） |
| c6  | concept       | Rule of Five（预告）                     | C++11 增加了移动构造和移动赋值。Rule of Five = Rule of Three + 移动构造 + 移动赋值。这里先建立概念——"资源管理类要同时管好拷贝和移动"，具体在 7.2 移动语义详讲                                 |
| c7  | practice      | 类管理堆内存但不写拷贝构造，会发生什么？ | choice: 编译错误 / 浅拷贝导致 double free / 自动变成深拷贝 / 什么也不发生 → 浅拷贝导致两个对象共享堆内存，析构时 double free                                                                  |

### 4.9 多态与虚函数 (7 卡)

> 扩充原设计（5→7 卡）。增加虚函数机制（vtable 简述）和纯虚函数/抽象类深入。多态是 OOP 四大支柱中最难的一个，7 卡不能再少。

| #   | 卡类型   | 标题                        | 内容要点                                                                                                                                                                                                                         |
| --- | -------- | --------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| c1  | concept  | 多态是什么                  | 同一个接口，不同行为。`Animal* a = new Dog(); a->speak();` — 指针是 Animal 类型，但调用的是 Dog 的 speak。多态 = 编译时按基类写，运行时走子类版本                                                                                |
| c2  | concept  | 虚函数与动态绑定            | `virtual` 让函数调用在**运行时**根据实际对象类型决定（动态绑定）。不写 virtual → **编译时**按指针/引用类型决定（静态绑定），永远调基类版本。**这是学生第一坑**——"为什么我重写了但没生效？"                                       |

| c4  | code     | 多态实战                    | 基类指针数组：`Animal* a[] = {new Dog(), new Cat(), new Bird()}; for(auto p : a) p->speak();` — 同一个 `p->speak()`，不同对象不同行为                                                                                            |

### 4.10 设计原则 (5 卡)

> 新增节点，OOP 收束。教完"怎么写"之后必须教"什么时候该用"——这是区分"会用 C++"和"会设计 C++ 程序"的分水岭。

| #   | 卡类型   | 标题                     | 内容要点                                                                                                                                                                                                                |
| --- | -------- | ------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| c1  | concept  | 组合优于继承             | 继承是 C++ 最被滥用的特性。"能用组合就别继承"——组合更灵活（运行时换组件）、耦合更低（只依赖接口）、不会被迫继承不需要的东西。反例：`class Stack : public vector<int>` 继承了不必要的 `operator[]`，破坏了栈的 LIFO 语义 |
| c2  | concept  | 面向接口编程             | 依赖抽象类（接口），不依赖具体类。`void drawAll(vector<Shape*>& shapes)` — 只依赖 Shape 接口，不关心每个 Shape 怎么画。新增子类不需要改 drawAll——这就是多态的真正价值                                                   |
| c3  | code     | 策略模式实例             | 不用继承表达行为差异：`class Sorter { SortStrategy* strategy; public: void sort(vector<int>& v) { strategy->sort(v); } };` — 排序算法运行时切换，继承做不到这一点                                                       |
| c4  | concept  | OOP 不是银弹             | 不是所有问题都适合用继承建模。过度设计 = 为"万一"写三层继承 = 代码变复杂、编译变慢。三问自查：① 真有变体吗？② 接口真的一样吗？③ 用组合是不是更简单？简单问题用 struct + 普通函数就是最佳方案                            |
| c5  | practice | 判断：该用继承还是组合？ | 情景选择：汽车和引擎→组合（has-a）；形状和圆→继承（is-a）；栈和 vector→组合（栈不是 vector，只是用它实现）。让学生理解判断标准                                                                                          |

## 设计理念与分析

### 模块排列逻辑

8 个模块按"用 → 理解 → 造"三层递进：

```
01 基础         →  "用"          语法/控制流/函数/数组   ← 能用 C++ 写程序
02 进阶         →  "理解底层"    指针/内存四区/struct     ← 知道程序在内存中怎么跑
                + struct(→OOP)   递归                    + OOP 和算法的引桥
03 面向对象     →  "自己造类"    分文件/封装/继承/多态   ← 从使用者变成设计者
04 流与文件     →  "用别人的类"  fstream/stringstream    ← 学完类，再看标准库类
05 STL          →  "用标准库"    vector/map/algorithm    ← 生产力跃升
06 泛型         →  "造通用工具"  模板/特化/concept       ← 写可复用的代码
07 现代 C++     →  "跟上前沿"    lambda/移动/Ranges      ← C++11/17/20 关键特性
08 工程化       →  "交付工程"    CMake/GDB/Shell/规范    ← 从能写变成能交付
```

关键设计决策：

- **03 OOP 放在 04 流之前**：学生先懂 class、构造析构、RAII，再去看 ifstream/ofstream——知道构造函数打开文件、析构自动关闭是 RAII 在工作，而不是"反正这么写就对了"。从模糊的模仿变成有理论支撑的理解。
- **02 进阶插入 struct 和递归**：struct 是 OOP 的自然引桥（和 class 只有默认权限的区别），打破 7 个连续指针节点的密度。递归承接内存四区中"栈帧"的概念——让栈知识落到具体算法上。
- **分文件编写放在 03-oop 开头**：class 天然 `.h` + `.cpp`，学了分文件立刻用于写类，即学即用。
- **06 泛型紧接 05 STL**：STL 大量使用模板（vector`<int>`、map`<string,int>`），学生对模板"用"熟了，再学"怎么写"模板，顺势切入。
- **01.12 数组末尾加 vector 预览卡**：只播种不展开。学生知道前面有更好的选择，先踏实学数组理解底层。
- **07 智能指针已提前在 02-进阶讲**：动态内存管理是进阶模块的核心主题，学完堆/栈/new/delete 立刻引入 unique_ptr/shared_ptr 作为"不泄漏的保险"。07-modern 不再重复。

## 架构约定：防止导入链单点故障

### 问题

所有课程节点通过 3 层静态 `import` 串联：`courses/index.ts` → `cpp/index.ts` → 模块 index.ts → 节点文件。任意一个节点文件有语法错误，整个 `courses` 数组导出失败，所有课程页崩溃。

### 解决方案

课程数据通过 `src/lib/useCourses.ts` 访问层暴露，不再直接 import `courses`：

```
屏幕组件 → useCourse(id) / useCourses()  ←  React hook
工具函数 → getCourse(id) / getCourses()  ←  同步函数
          ↓
    src/lib/useCourses.ts（唯一访问点）
          ↓
    src/data/courses/index.ts（静态导出）
```

**规则**：
- 所有消费方通过 `@/lib/useCourses` 获取课程数据，**禁止**直接 `import { courses } from '@/data/courses'`
- 新增第二门课程时，只需改 `useCourses.ts` 内部实现——将静态 import 改为动态 `import()` + try-catch，对外 API 不变
- `validate.test.ts` 是编译期的安全网——提交前运行即可捕获断裂的 import 链

### 影响范围

数据层 `courses/index.ts` 仍是静态导出。当切换到多课程动态加载时，8 个消费方屏幕不需要任何改动——只改 `useCourses.ts` 一个文件。
