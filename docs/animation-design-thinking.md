# 动画设计思路

> ScopeCodePlayer 从失败到成功的迭代过程，提炼为通用设计原则。

## 核心教训：先问"这个动画到底要展示什么关系"

MemoryBox 擅长展示**"内存里有什么"**——格子大小代表类型宽度、颜色区分变量、地址列展示位置。但作用域的核心不是"内存里有什么"，而是**"代码执行到这一行时，哪些变量还活着"**。

| 概念 | 本质关系 | 适合的工具 |
|------|---------|-----------|
| 变量内存布局 | 变量 ↔ 内存地址 | MemoryBox |
| 变量作用域 | **代码位置 ↔ 变量存活状态** | ScopeCodePlayer |
| 条件分支 | 条件值 ↔ 执行路径 | ？ |

**第一原则：不是所有可视化都适合用 MemoryBox。先想清楚这个概念的本质关系是什么。**

## 三轮迭代的过程

### 第一版：MemoryBox — 变量出现 → 消失

```
Step 0: 空 → Step 1: a → Step 2: a+b → Step 3: a → Step 4: 空
```

**被拒原因**：和已有的 variable-storage 动画没区别。学生看到的是一堆格子出现又消失，但不知道为什么。代码和执行位置完全不在画面里。

### 第二版：MemoryBox + 灰格子 — 活着/死了区分

```
Step 0: 空 → Step 1: a(绿) → Step 2: a(绿)+b(蓝) → Step 3: a(绿)+b(灰) → Step 4: 全灰
```

**被拒原因**：灰格子虽然区分了生死，但学生仍然不知道"为什么 b 死了？是执行到哪一行导致它死的？"问题从"缺少状态区分"升级为"缺少因果链条"。

### 第三版：ScopeCodePlayer — 代码 + 箭头 + 内存同步

```
代码区（当前行高亮 + ← 箭头）
    ↓
内存格子（绿=活着 / 灰=已死）
    ↓
注释（解释当前发生了什么）
```

**通过**：三要素建立完整因果链——**代码位置（因）→ 箭头（连接）→ 内存状态（果）→ 注释（解释）**。

## 动画设计决策链

写一个新动画前，按顺序回答四个问题：

### 1. 这个概念的"因"和"果"分别是什么？

| 概念 | 因（什么触发了变化） | 果（什么改变了） |
|------|-------------------|-----------------|
| 作用域 | 代码执行到 `{` / `}` | 变量的存活状态改变 |
| 指针赋值 | `p = &x;` 执行 | p 的值变成 x 的地址 |
| 条件分支 | 条件表达式求值 | 选择不同的执行路径 |

**因和果都要在画面上能看到**。如果动画只展示果没展示因，学生会问"为什么"。

### 2. "因"能不能可视化？

- 代码执行位置 → **可以**：高亮代码行（ScopeCodePlayer 的方案）
- 用户输入了一个值 → **可以**：标注"用户输入了 5"
- 编译器推导了一个类型 → **部分**：加文字说明
- nullptr 解引用 → **不能**：运行时崩溃不适合动画

**因如果不能可视化，这个动画就不适合做**。

### 3. MemoryBox 够不够？

| 场景 | 够 |
|------|:---:|
| 展示变量的字节大小差异 | ✓ |
| 展示数组的连续内存布局 | ✓ |
| 展示栈变量 vs 堆变量的位置 | ✓ |

| 场景 | 不够，需要新组件 |
|------|:---:|
| 展示代码行和内存状态的对应关系 | ✗ → ScopeCodePlayer |
| 展示程序执行的两种分支路径 | ✗ → 需要分支对比组件 |
| 展示循环的多次迭代 | ✗ → 需要循环展开组件 |

### 4. 新组件能不能复用？

ScopeCodePlayer 的设计保证了复用性：
- `sourceCode` 是字符串，换一套代码即可
- `highlightLines` 是数字数组，指向不同行
- `allocations` 沿用 `VarAlloc`，语法和 MemoryBox 完全一样
- `annotation` 是纯文本，每步不同

**一个新组件 = 一个可复用的模板**。写的时候不要把特定课程内容写死在组件里。

## 颜色语义规范

ScopeCodePlayer 确立了颜色编码规则，后续新动画应保持一致：

| 颜色 | hex | 语义 |
|------|-----|------|
| 绿 `#2ed573` | 第一个变量，活着 |
| 蓝 `#4a9eff` | 第二个变量，活着 |
| 灰 `#666666` | 已销毁 / 离开作用域 |
| 橙 `#ff9f43` | 第三个变量 / 特殊标记 |
| 紫 `#a55eea` | double / 大类型 |

**每个颜色有固定的语义，不随机选色。** 绿色永远是"活着"，灰色永远是"死了"。

## 回顾 1.5 的教学原则在这里仍然适用

```  
判断标准：这个变量还活着吗？
    → 看代码执行到哪一行，在当前 {} 里面就是活着

对照例子：
    ✓ 内层 {} 里能访问外层 a（两个都绿）
    ✗ 离开内层 {} 后 b 就死了（变灰）

行为总结：
    绿色 = 活着，灰色 = 死了。{} 决定生死。
```

动画不是替代教学，是**把教学原则变成视觉**。
