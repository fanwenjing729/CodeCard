# 课程内容创作指南

> 抽象概念教学原则 + 卡片设计自检清单。写新节点前过一遍。

## 核心原则

**给新手讲抽象概念时，只给定义不够，必须给三样东西：**

### 1. 判断标准 — 一个能直接套用的问题

学生需要的不只是"XX 是什么"，而是"我遇到 XX 时怎么判断"。

把抽象概念翻译成一个**二选一的问题**，让学生能对任何代码场景直接套用。

| 概念 | 不要只讲定义 | 要给学生一个判断问题 |
|------|------------|-------------------|
| const vs constexpr | "constexpr 是编译期常量" | "编译器读到这一行时，能不能直接算出这个值？" |
| 编译期 vs 运行期 | "编译期先于运行期" | "这个值写在源码里就能确定，还是要等用户输入？" |
| 变量作用域 | "变量在其声明的块内可见" | "这行代码在不在变量的那对大括号里面？" |
| lvalue vs rvalue | "左值有地址右值没有" | "能不能对它取地址 &？" |

### 2. 对照例子 — 能 ✓ 和不能 ✗ 各一个

绝对不要只给"能"的例子，必须同时给"不能"的例子。

```
错误示范：
  constexpr int MAX = 100;  // OK！

学生看完觉得自己懂了。然后写 constexpr int y = cin >> x; 编译错误，不知道为什么。

正确示范：
  例子 1 — 能 ✓：
    constexpr int MAX = 100;  // 100 就写在源码里，编译时直接确定

  例子 2 — 不能 ✗：
    int x; cin >> x;
    constexpr int y = x;  // 编译错误！x 值要等用户输入才知道
```

两个例子摆在一起，学生自己就能看出规律。规律自己发现的 > 你告诉他的。

### 3. 行为总结 — 一句话记住它

每个概念收尾给一句"人话"，不超过 20 个字，能直接记在脑子里。

```
const     → "初始化后不能改"
constexpr → "编译时必须算出来"
#define   → "纯文本替换，别用"
```

**检测标准**：这句话能不能让一个没看过前面内容的同学，看完后大概明白这概念在说什么？能 → 过了。不能 → 太长/太抽象，重写。

## 卡片粒度自检

写完后对每张概念卡过这三个问题：

1. **一张卡只聚焦一个概念？** — 如果一张卡里塞了 const + constexpr + #define 三个东西，拆。每张卡应该能用一个短语概括（"什么是编译期"、"#define 为什么不用"）。

2. **对照例子有了吗？** — 如果只有"正确用法"没有"错误用法会导致什么"，补。新手需要看到错误的后果。

3. **术语落地了吗？** — "编译期"、"预处理"、"未定义行为" 这些词新手不懂。如果非用不可，先用一句话解释。更好的做法是先给具体例子，再引出术语。

## 常见陷阱

| 陷阱 | 表现 | 修正 |
|------|------|------|
| 术语堆砌 | "constexpr 声明的是编译期求值的常量表达式" | 换成："constexpr 要求值在编译时就能确定。比如 100 写在源码里，编译时能看到，OK。cin >> x 等用户输入，编译时不知道，不行。" |
| 超前引用 | 在讲 A 时用了还没讲的 B（如讲变量时用 const） | 删掉或注释掉，等 B 讲完再回来补 |
| 信息过载 | 一张概念卡塞了 5 个知识点 | 拆，一张卡一个概念 |
| 只有正例 | 代码示例全是对的，没有错的 | 补一个"如果写错会怎样" |
| 表格用空格对齐 | 依赖等宽字体 | 改用编号列表 |

## 写新节点的流程

1. 看设计文档确定卡片数量和类型
2. 对每张 concept 卡：确定**一个**核心概念 → 想好判断问题 → 准备两个对照例子 → 写一句话总结
3. 对每张 code 卡：代码要能跑（有 main + include），不引用还没讲的概念
4. 对每张 practice 卡：选项要有迷惑性（常见错误作为干扰项），解析要讲"为什么"
5. 写完回来看：有没有哪个抽象术语没用具体例子解释？有没有超前引用？
